# API 与运维

HTTP 请求与响应字段见 [API 参考](api.md)，部署步骤见 [部署指南](deployment.md)。

## 正式请求流程

1. 客户端创建匿名玩家并仅保存访问令牌；恢复码只展示一次。
2. 客户端读取服务端发布的北京时间当日题目。
3. 开局时 API 创建 15 分钟有效的尝试并签发一次性 JWT。
4. 客户端提交移动序列、单调活动时间和操作日志，不提交可信星级。
5. API 按共享规则重放路径，以活动时长排名，并用服务器墙钟验证活动时长上限。
6. 有效成绩先取得玩家+日期事务锁，再原子更新每日最佳并从每日最佳重算月度赛季聚合。

## 监控建议

- 定时检查 `/health`、API p95 延迟、5xx 比例和 PostgreSQL 连接数。
- 对 `attempts.valid = false` 的比例与 `rejection_reason` 分布设置告警。
- 每日检查下一日谜题是否可生成；当前实现也会在首次读取时确定性生成并持久化。
- 每次发布后运行 `scripts/postgres-smoke.sql`；持续监控 API `/health` 和 Compose 健康状态。
- 数据库每日执行自定义格式备份，至少保留 14 天，并每月演练一次恢复。

## 安全边界

- `JWT_SECRET` 至少使用 32 字节随机值，禁止提交到仓库。
- 生产环境的 `ALLOWED_ORIGIN` 只填写实际 GitHub Pages 或自定义域名。
- 恢复码只保存 SHA-256 摘要，使用成功后立即撤销并轮换。
- 本实现提供实用级防作弊；共享答案或自行求解无法仅靠客户端日志阻止。

## 数据保留

- `puzzles` 和榜单数据用于历史赛季查询，应长期保留。
- `attempts` 包含完整路径和操作日志，可按活跃量设置保留周期，例如 90 天。
- 清理尝试前必须保留 `daily_best_scores` 引用的记录，或先调整外键与审计策略。
- 已撤销恢复凭据可定期归档，当前有效摘要必须保留。

## 故障处理

### 每日题无法生成

生成器会使用已验证备用题。检查 `failed to prepare daily puzzles` 日志以及数据库连接；不要修改已经发布的当日定义。

### 成绩提交失败率升高

按 `rejection_reason` 区分路径无效、日志不完整、时间异常和尝试过期。若集中发生在新客户端版本，检查 `rulesVersion` 与 `minClientVersion`。

### 赛季聚合异常

赛季表可从 `daily_best_scores` 重建。修复前先备份数据库，再按玩家和月份重新执行聚合。

### 恢复码丢失

匿名身份没有人工证明所有权机制。本地令牌与恢复码同时丢失后，不能安全找回原账号。
